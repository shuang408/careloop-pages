Module 01

模块一：API / Token / 聚合平台

先从客户最容易感知的“调用入口、计量方式、统一平台”开始。

这一模块解决一个最基础的问题：客户到底怎么使用大模型？平台怎么知道谁在用、用了多少、该扣多少钱、调用失败怎么办、多个模型怎么统一管理。你可以把它理解成 AI 基础设施的“门面 + 收银台 + 调度台”。

1. 三个核心概念先分清

API 是客户调用模型能力的入口。客户的系统不会每次打开网页聊天，而是通过 API 把问题发给模型，然后拿回结果。
Token 是模型处理文本时的计量单位。输入内容、系统提示词、历史上下文、知识库片段、模型输出都会产生 token。
聚合平台 是统一管理多个模型、多个客户、多个 API Key、多个账单、多个权限和多个风控规则的平台。

通俗类比：API 像餐厅点餐窗口；Token 像水表电表；聚合平台像商场服务台 + 收银系统 + 调度系统。

2. 一次 API 请求里面有什么

一次 API 请求通常包含：请求地址、API Key、模型名称、用户输入、系统提示词、历史消息、参数设置、是否流式输出等。平台收到请求后，会先判断“你是谁”“有没有权限”“余额够不够”“调用是否超限”，再决定调用哪个模型。

常见参数包括：temperature 控制回答随机性，max_tokens 控制最大输出长度，stream 控制是否流式返回，tools 控制是否允许模型调用工具。

3. API Key 为什么重要

API Key 是调用凭证。它不只是一个字符串，而是客户身份、项目权限、额度限制、日志审计和风险控制的入口。

一个成熟平台应该支持：创建 Key、删除 Key、限制 Key 可调用模型、限制 Key 每日额度、绑定 IP 白名单、查看 Key 调用日志、检测异常调用。

注意点：API Key 泄露会造成真实费用损失，所以企业客户很关心 Key 权限、预算上限和审计日志。

4. Token 成本到底怎么产生

Token 成本不是只看用户问了几个字。一次请求的输入 token 包括：系统提示词、用户问题、历史上下文、知识库检索内容、工具调用结果。输出 token 是模型生成的答案。

为什么长文档贵？因为文档全文进入上下文，会产生大量 input token。为什么 Agent 贵？因为 Agent 往往要多次调用模型，每一步都可能累积上下文和工具结果。为什么知识库也贵？因为检索出来的资料会被塞回模型上下文。

5. 聚合平台解决什么问题

如果客户自己分别接 OpenAI、Claude、Gemini、DeepSeek、通义、智谱等模型，会遇到接口不统一、价格不统一、Key 分散、日志分散、账单分散、故障无法自动切换的问题。

聚合平台把这些能力统一起来：统一 API、统一鉴权、统一计费、统一日志、统一模型路由、统一限流、统一内容安全、统一成本看板。客户只接你的平台，不需要关心背后用了哪个模型。

6. 模型路由是平台的核心能力

模型路由不是简单转发请求，而是根据场景、成本、速度、质量和稳定性选择模型。

普通问答可以走低成本模型；复杂推理走强模型；代码任务走代码模型；长文档走长上下文模型；图片任务走多模态模型；某个供应商失败时自动切换备用模型。

平台价值会从“模型多”升级为“帮客户在效果、成本、速度、稳定性之间做选择”。

7. 你需要盯住的指标

API 平台的关键指标包括：input tokens、output tokens、total tokens、RPM、TPM、QPS、TTFT、Latency、Error Rate、Timeout Rate、Cost per Request、Revenue per Request、Gross Margin、Fallback Rate。

产品视角看：客户体验主要看延迟和成功率；经营视角看单次请求成本和毛利；运维视角看错误率、超时率和备用模型切换比例。

8. 常见误区

误区一：以为卖 token 就是倒卖官方 token。更准确地说，是提供可计费、可管理、可风控的 AI 能力服务。
误区二：以为接几个模型 API 就是聚合平台。真正的平台还要有账户、权限、日志、账单、限流、风控、路由、成本看板。
误区三：只看模型价格，不看重试、日志、工具调用、内容审核、客服和运维成本。

9. 基础补充：API 平台到底像什么

你可以把 API 平台想成“高速公路收费站”。每辆车进入高速时要识别车牌，这就是 API Key；要判断能不能上高速，这就是权限；要知道开了多少公里，这就是 token 计量；要根据拥堵情况选择路线，这就是模型路由；最后要扣费和生成账单，这就是计费系统。

10. 平台为什么必须有日志

日志不是给技术人员看的流水账，而是客户信任的证据。客户会问：哪个部门用了多少钱？哪个 API Key 调用了哪些模型？哪次调用失败了？为什么扣了这么多？没有日志，就无法解释账单，也无法追责和排查故障。

11. Token 预算怎么设计

企业平台要支持按企业、项目、成员、API Key 设置预算。比如企业每月 10 万元，研发部门 4 万，客服部门 3 万，市场部门 2 万，测试项目 1 万。预算不是财务功能，而是防止成本失控的核心产品能力。

12. 聚合平台 MVP 最小闭环

最小闭环不是“接入很多模型”，而是：客户能开户、能创建 API Key、能调用模型、能统计 token、能扣费、能查日志、能看账单、能限制额度、能在失败时返回明确错误。先把这条链跑通，再做更复杂的模型路由。

13. 你以后看 API 产品要看什么

看文档是否清晰、SDK 是否完善、错误码是否完整、是否支持流式输出、是否有速率限制说明、是否能导出账单、是否有企业权限、是否有安全策略、是否有模型降级和备用方案。

API 平台常见角色

开发者关心接口好不好接、延迟低不低、错误码清不清楚。
企业管理员关心成员权限、预算、账单、审计和风险控制。
财务关心充值、发票、对账、部门分摊和合同。
运维关心成功率、延迟、故障、告警和备用模型。
平台经营者关心成本、毛利、模型调用占比和客户留存。

模块二：模型部署与推理服务

理解开源模型如何真正跑起来，并变成可调用的服务。

1. 训练、微调、推理不要混在一起

训练是从大量数据中学习模型参数，资源消耗最大。微调是在已有模型基础上用行业数据适配特定任务。推理是模型根据用户输入生成答案。

AI 基础设施商业化里，最常遇到的是推理服务：客户已经有模型或选择开源模型，希望你把它部署成稳定、可访问、可计费的 API。

2. 模型权重是什么

模型权重可以理解为模型的大脑参数。开源模型通常会发布权重文件，部署模型就是把这些权重加载到 GPU 显存中，再由推理框架运行。

模型越大，权重占用显存越多。上下文越长，KV Cache 占用越高。并发越高，同时存在的缓存和请求越多，显存压力越大。

3. 推理框架的作用

推理框架负责把模型权重变成可访问服务。它要做模型加载、请求排队、batch 合并、KV Cache 管理、显存优化、流式输出、API 暴露。

常见工具包括 vLLM、TensorRT-LLM、TGI、Ollama。学习阶段可以先理解 vLLM：它适合高吞吐 LLM 服务，并且常见于私有模型 API 部署。

4. KV Cache 为什么重要

大模型生成文本时，会保存前面 token 的 key/value 信息，避免每次都重新计算，这就是 KV Cache。它能提升生成效率，但会消耗显存。

长上下文、多轮对话、高并发都会增加 KV Cache 压力。很多推理服务成本和容量规划，核心就是在管理 KV Cache。

5. 量化是什么

量化是降低模型数值精度，从而减少显存占用和计算成本。例如从更高精度压到 8bit 或 4bit。

优点：可以让更大模型跑在更小显存上，降低成本。风险：可能影响效果、稳定性或某些复杂任务表现。产品上不能只说“能跑”，还要评估“效果是否能接受”。

6. 标准部署链路

选择模型 → 获取权重 → 准备 CUDA / Python / 依赖环境 → 选择 GPU 服务器 → 选择推理框架 → 启动模型服务 → 暴露 API → 接入聚合平台 → 接入日志、计量、监控、告警。

这条链路说明：模型部署不是算法问题，而是模型、系统、硬件、网络和运维共同作用的工程问题。

7. 推理服务关键指标

TTFT 是首 token 延迟，决定用户多久看到第一个字。TPOT 是每个输出 token 的生成速度。Latency 是总延迟。Throughput 是单位时间处理能力。QPS 是每秒请求数。GPU Utilization 代表 GPU 利用率。VRAM Usage 代表显存占用。

用户体验看 TTFT 和 Latency；平台经营看 Throughput 和 GPU 利用率；运维稳定看错误率、超时率、OOM。

8. 常见故障

OOM：模型太大、上下文太长或并发太高。响应慢：请求排队、GPU 利用率高、batch 不合理。服务超时：输出太长、实例不足、网关超时设置短。环境跑不通：CUDA、驱动、Python 包版本冲突。成本过高：模型过大、利用率低、缺少路由和缓存策略。

9. 基础补充：部署模型和调用模型的区别

调用模型是“别人已经把模型部署好了，你通过 API 使用”。部署模型是“你自己准备服务器、GPU、环境、模型文件和推理框架，把它变成可调用服务”。前者更像使用网约车，后者更像自己买车、保养、雇司机、做调度。

10. 为什么推理服务要接入网关

单独启动一个模型 API 只能说明模型能回答问题，但不能说明它能商业化。接入网关后，才能统一 API Key、权限、限流、token 统计、日志、账单、内容安全和 SLA。没有网关，模型服务只是技术 demo。

11. 私有化部署最容易漏什么

很多人只关注“模型能不能跑”，但私有化更容易漏：显卡驱动版本、CUDA版本、内网离线安装、模型权重存放、日志脱敏、客户权限、备份策略、升级方案、故障响应、硬件备件、验收标准。

12. 推理成本优化思路

优化推理成本通常有几类：选择更小模型、量化模型、限制上下文长度、优化 prompt、做缓存、批处理、模型路由、冷热实例分层、提高 GPU 利用率、把高峰和低谷任务错峰。

13. 什么时候不建议自部署

如果客户调用量很低、没有数据敏感要求、没有专属低延迟要求、也没有运维能力，自部署可能比直接调用成熟模型 API 更贵。自部署的价值通常来自数据安全、成本规模化、模型可控、专属性能或合规要求。

推理服务验收清单

功能模型是否能按预期回答，是否支持流式输出和工具调用。
性能TTFT、总延迟、QPS、并发、吞吐是否达到要求。
稳定性长时间运行是否崩溃，OOM、超时、重启频率如何。
安全权限、日志、数据隔离、内网访问是否满足客户要求。
运维是否有监控、告警、备份、升级和故障响应机制。

1. 为什么 AI 更依赖 GPU

大模型推理和训练涉及大量矩阵计算，GPU 擅长并行计算，所以比 CPU 更适合。对于大模型来说，GPU 不仅提供算力，还提供显存。很多时候不是算力不够，而是显存放不下模型和上下文。

2. GPU 服务器看哪些参数

关键参数包括：GPU 型号、单卡显存、GPU 数量、GPU 间互联、CPU、内存、NVMe 存储、网络带宽、电源功率、散热方式。

入门理解：GPU 是计算核心，显存是模型和上下文的工作空间，网络影响多机训练和数据传输，存储影响模型加载和数据集读写。

3. 显存和模型的关系

模型权重会占显存，KV Cache 会占显存，并发请求会占显存，框架运行也会有额外开销。

模型越大，需要越多显存；上下文越长，KV Cache 越大；并发越高，同时保留的缓存越多。如果显存不足，就会 OOM 或只能降低并发、降低上下文、量化模型或多卡部署。

4. 算力资源的交付形态

裸金属 GPU：客户独占整台服务器，性能强、隔离好。GPU 云主机：像云服务器一样按规格创建实例。容器实例：更轻量，适合任务化和平台化。推理实例：直接给客户模型 API，不暴露底层机器。私有化一体机：硬件、模型、平台一起交付到客户现场。

5. GPU 租赁客户在买什么

训练团队买的是训练时间和多卡性能；AI 应用团队买的是推理吞吐和稳定性；企业客户买的是数据安全和可控；开发者买的是便宜、快速、环境好；科研团队买的是可复现实验环境和算力弹性。

6. 利用率是经营核心

GPU 最大成本不是买的那一刻，而是长期折旧、电费、机柜费、带宽费、运维费。GPU 空着也会产生成本。

所以经营上最重要的是利用率：GPU 被客户实际使用的时间越高，收入越稳定；空置率越高，亏损压力越大。

7. 资源碎片问题

如果客户租 1 张卡、2 张卡、4 张卡、8 张卡，平台会出现不同大小的空闲碎片。某些节点剩余 1 张卡，可能无法满足 4 卡任务；某些任务长期占用但利用率低，会影响整体收益。

这就是为什么后面要学集群调度和资源池。

8. 采购和方案判断

不要只问“买什么 GPU”。应该先问：客户是训练还是推理？模型多大？并发多少？上下文多长？是否需要多机？数据量多大？是否长期使用？预算多少？是否需要私有化？这些问题决定硬件方案。

GPU资源指标

GPU型号决定算力、生态和成本
显存决定模型能否放下、上下文能多长
GPU数量决定并发、训练规模和多卡能力
网络影响多机训练和数据传输
存储影响模型加载和数据集读写
功耗散热影响机房电力和制冷要求

9. 基础补充：显存不是硬盘

很多人会把显存和硬盘混淆。硬盘用来长期保存模型文件和数据集，显存是 GPU 运行模型时的高速工作空间。模型文件存在硬盘上不等于模型能跑，必须能加载进显存并留出运行空间。

10. 为什么同样 8 张 GPU，效果可能不同

不仅看 GPU 数量，还要看单卡显存、GPU间互联、服务器架构、网络、存储、散热和软件优化。8 张低显存卡不一定能替代 8 张高显存卡；多卡之间通信慢，也会拖累训练效率。

11. GPU 云报价为什么差异大

差异来自 GPU 型号、显存、是否独享、是否裸金属、带宽、存储、机房位置、是否含技术支持、是否包月、是否有 SLA、资源是否紧缺。看价格时要看总成本，不只是小时单价。

12. 资源利用率怎么影响商业模式

如果平台买了 100 张 GPU，每天只有 30 张被客户使用，那么其余 70 张仍然产生折旧、电费、机柜费和运维费。高利用率能摊薄成本，低利用率会让看似高毛利的业务变成亏损。

13. 客户选择 GPU 时常见误区

误区包括：只看 GPU 型号不看显存；只看小时价格不看稳定性；只看卡数不看互联；只看能不能开机不看环境；只看短期价格不看长期运维；以为训练和推理需要同样配置。

训练、推理、开发测试的资源差异

开发测试更关注便宜、快速启动、镜像好用、Notebook方便。
推理服务更关注稳定、低延迟、显存、并发、SLA和成本。
微调更关注显存、数据读取、训练框架、检查点保存。
大规模训练更关注多卡、多机、高速网络、存储吞吐和容错。

15. 如何估算“这张卡能不能跑这个模型”

最粗略的估算方法是先看模型权重占多少显存：FP16/BF16 通常约等于“参数量 × 2字节”。所以 7B 模型权重大约 14GB，14B 大约 28GB，32B 大约 64GB，70B 大约 140GB。

但模型运行不是只放权重，还要给 KV Cache、batch、框架、CUDA、通信缓冲等留空间。推理时建议至少预留 20% 到 40% 的显存余量；长上下文和高并发时，余量要更多。

量化可以显著降低显存占用：8-bit 大致接近参数量 × 1字节，4-bit 大致接近参数量 × 0.5字节，再加量化元数据和运行开销。比如 70B 模型 4-bit 可能在 40GB 到 55GB 左右起步，但如果上下文很长、并发很高，80GB 显存也可能很快吃紧。

16. GPU 选型口诀

只做学习和本地实验：看预算和显存，RTX 4090 / 5090 这类消费级卡可以入门，但不适合严肃数据中心托管。

做轻量推理或视频/Embedding：看功耗、成本和密度，L4 这类低功耗卡更适合规模化小模型服务。

做企业中等模型推理、图像生成、开发测试：看 48GB 级别显存，L40S、RTX 6000 Ada 这类更均衡。

做大模型训练、高吞吐推理、私有化大客户：看 H100、H200、B200、MI300X、MI325X 这类数据中心卡，还要看多卡互联、网络、机房电力和推理框架。

最终不要问“哪张卡最好”，而要问：模型多大、精度要求、上下文多长、并发多少、是训练还是推理、是否多租户、预算和 SLA 要求是什么。

模块四：集群调度与云平台层

把多台 GPU 服务器组织成可出租、可管理、可计费的资源池。

这一模块解决“多台 GPU 服务器怎么变成云平台”的问题。单台机器可以人工管理，多台机器必须资源池化、容器化、自动调度、监控、计费和权限隔离。

  5. 5. 云平台控制台做什么

1. 为什么需要集群调度

当只有一台机器，可以手动登录、装环境、启动服务。当有几十台 GPU 服务器时，手工管理会变成灾难：不知道哪台空闲、谁在用、用了多久、故障在哪里、费用怎么算。

集群调度的目标是把分散服务器变成统一资源池，让客户自助创建实例，系统自动分配资源。

2. Docker 和镜像的作用

Docker 解决环境一致性。AI 环境复杂，可能包含 CUDA、驱动、Python、PyTorch、Transformers、vLLM、Jupyter。镜像把这些环境打包好，客户创建实例时直接选择镜像，不用从零配置。

3. Kubernetes 基础概念

Cluster 是集群，Node 是节点，Pod 是最小部署单元，Container 是容器，Deployment 管理长期运行服务，Job 管理一次性任务，Service 提供稳定访问入口，Namespace 做逻辑隔离，Scheduler 决定任务放到哪个节点。

4. GPU 调度怎么发生

客户选择 GPU 规格，平台生成任务描述，调度器查找可用节点，设备插件把 GPU 暴露给 Kubernetes，容器获得 GPU 访问权限，任务开始运行。

调度不只看“有没有 GPU”，还要看 GPU 型号、显存、节点负载、网络、客户等级、任务类型和资源碎片。

5. 云平台控制台做什么

客户不会直接操作 Kubernetes。客户看到的是控制台：创建实例、选择镜像、选择 GPU、挂载存储、设置网络、查看监控、查看账单、提交工单。

底层复杂度被平台隐藏，客户只需要理解“我选什么配置、用多久、花多少钱、哪里看日志、怎么释放”。

6. 资源生命周期

下单 → 权限和余额检查 → 资源匹配 → 调度创建 → 拉取镜像 → 绑定 GPU/存储/网络 → 运行监控 → 计量扣费 → 释放回收。

这条生命周期连接了产品、技术、运维、计费和客户体验。

7. 多租户隔离

云平台通常服务多个客户，所以要隔离账号、项目、容器、网络、存储、日志和账单。常见能力包括企业空间、项目空间、角色权限、Namespace、网络策略、IP 白名单、审计日志和资源配额。

8. 监控和告警

必须监控 GPU 利用率、显存、温度、CPU、内存、磁盘、网络、容器状态、重启次数、任务排队时间、客户用量和空置率。没有监控，就不知道资源是否健康，也无法做 SLA。

云平台功能

实例管理创建、启动、停止、释放GPU实例
镜像市场提供CUDA、PyTorch、vLLM等环境
资源调度把任务放到合适节点
监控面板展示GPU、显存、网络、日志
账单中心记录时长、规格、存储、流量
权限隔离企业、项目、成员、多租户

9. 基础补充：资源池是什么

资源池就是把很多台服务器抽象成一个统一池子。客户不需要知道具体是哪台机器，只需要选择“我要 1 张 GPU、80GB 显存、PyTorch 镜像、运行 10 小时”，平台自动找到合适节点并启动环境。

10. 为什么客户不直接用 Kubernetes

Kubernetes 是底层编排工具，复杂且偏工程。客户真正需要的是产品化控制台：点选配置、创建实例、查看监控、管理数据盘、看账单、提交工单。平台把复杂底层能力包装成简单界面。

11. 镜像市场为什么关键

AI 环境安装复杂，镜像市场可以提供 CUDA、PyTorch、TensorFlow、vLLM、Jupyter、训练框架等标准环境。镜像越完善，客户启动任务越快，售后问题越少。

12. 多租户为什么难

多个客户共享资源池时，必须隔离账号、网络、存储、容器、日志、账单和权限。任何隔离失败都可能造成数据泄露、越权访问或账单混乱。

13. 调度策略不是只看空闲

调度要考虑 GPU 型号、显存、客户等级、任务优先级、节点健康、镜像缓存、数据位置、网络带宽和资源碎片。好的调度能提高利用率，差的调度会造成资源浪费和客户排队。

云平台后台发生了什么

客户点击创建前端提交规格、镜像、存储、网络、时长。
订单检查检查权限、余额、配额、资源是否可用。
调度资源找到合适节点，绑定 GPU、CPU、内存和存储。
启动环境拉镜像、启动容器、开放端口、初始化服务。
持续运营监控状态、记录用量、扣费、告警、释放资源。

模块五：IDC 机房、电力、制冷、网络与运维

理解服务器最终放在哪里，怎样稳定运行。

这一模块解决“算力资源的物理底座是什么”的问题。GPU 服务器最终要放在机房里，机房提供机柜、电力、制冷、网络、安全和运维。AI 机房不是普通房间，而是高功率、高散热、高网络要求的工程系统。

1. IDC 是什么

IDC 可以理解为专业数据中心，给服务器提供稳定、安全、可扩展的运行环境。它不直接产生模型能力，但决定模型服务能不能长期稳定运行。

客户感受到的是 API 是否稳定、GPU 实例是否可用；机房层面真实发生的是供电、散热、网络、消防、安防和巡检。

2. 机柜和 U 位

服务器安装在机柜里。U 位是服务器高度单位。但 AI 服务器不能只看还能放几个 U，更要看单柜功率、承重、走线、散热和网络接入。

AI GPU 服务器通常功耗高、重量大、线缆多，很多时候限制不是空间，而是电力和散热。

3. 电力系统

电力是 AI 机房第一约束。核心概念包括市电、UPS、不间断电源、柴油发电机、PDU、电源冗余、双路供电、机柜功率。

学习时要问：每柜多少 kW？是否支持高密 GPU？是否双路供电？UPS 冗余怎样？后续还有多少电力余量？

4. 制冷和液冷

服务器消耗的电大多转化为热量。制冷系统负责把热量带走。普通机房常见风冷，高密 AI 机房越来越关注液冷。

散热不足会导致 GPU 降频、故障、宕机、寿命下降。液冷适合高密度、高功率的 AI/HPC 场景。

5. 网络系统

网络分外部网络和内部网络。外部网络连接客户，包括公网、BGP、多线、专线、防火墙。内部网络连接服务器、存储和 GPU 集群，影响多机训练、模型加载和日志采集。

普通推理更关注客户访问延迟；大规模训练更关注内部高速网络。

6. 消防安防

机房设备价值高、风险高，必须有消防、门禁、监控、访客登记、区域权限、巡检制度。企业客户尤其关心物理访问控制和数据安全边界。

7. 运维监控

机房运维监控对象包括电压、电流、PDU负载、UPS状态、温湿度、液冷水温、网络丢包、服务器电源、风扇、磁盘、GPU温度、门禁、水浸、烟感、工单响应时间。

机房故障会一路传导到云平台和客户业务。

8. AI 高密机房特点

AI 机房和传统 IDC 的区别在于功率密度更高、散热压力更大、网络要求更强、运维难度更高、回本压力更强。AI 基础设施正在从单台服务器走向整机柜、液冷、高速互联、整体交付。

9. 基础补充：为什么机房不能随便找个房间

GPU 服务器需要稳定电力、强制冷、高速网络、消防安防、承重机柜和专业运维。普通办公室没有足够电力、散热和安全保障，服务器可能频繁宕机，甚至存在安全风险。

10. 电力为什么比机柜数量更重要

看似还有很多空机柜，但如果电力不够，就不能继续上 GPU 服务器。AI 服务器功耗高，经常是“空间够、电不够、冷不够”。所以问机房资源时要问每柜功率和电力余量。

11. 液冷为什么越来越重要

当单柜功率越来越高，单纯靠空气带走热量会很困难。液冷用液体更高效地带走热量，适合高密度 AI 机柜。但液冷也带来管路、维护、漏液风险和工程复杂度。

12. 网络不是只有公网带宽

公网带宽影响客户访问；内网影响存储、监控和服务调用；高速互联影响多机训练；专线影响企业客户连接；BGP影响多线路访问质量。不同业务对网络关注点不同。

13. 机房故障如何传导

机房电力异常可能导致节点掉线；制冷异常可能导致 GPU 降频或宕机；网络异常可能导致 API 超时；消防安防异常可能影响运维进入现场。最终客户看到的是服务不可用。

AI机房沟通时要问的问题

电力每柜多少 kW？是否双路供电？UPS和发电机如何配置？
制冷风冷还是液冷？支持多高功率密度？温湿度如何监控？
网络BGP、专线、公网带宽、内网带宽、交换机能力如何？
运维是否7×24值守？备件如何？故障响应时间多久？
合规消防、安防、资质、客户审计要求能否满足？
Module 06

模块六：计费、运营、交付、SLA 与合规

把技术资源变成可销售、可计费、可交付的服务。

这一模块解决“技术系统怎么变成商业服务”的问题。模型、GPU、服务器和机房只是资源；要真正运营，需要账户、权限、计量、计费、账单、发票、SLA、工单、风控、合规和经营看板。
本模块目录
  1. 1. 运营层的作用
  2. 2. 计量系统
  3. 3. 计费模型
  4. 4. 成本与毛利
  5. 5. 账户权限和预算
  6. 6. 交付形态
  7. 7. SLA 和工单
  8. 8. 合规和风控

1. 运营层的作用

技术层解决能不能用，运营层解决能不能卖、能不能算账、能不能长期服务、出了问题谁负责、客户敢不敢持续使用。

没有运营层，平台会变成内部工具；有运营层，才可能变成商业服务。

2. 计量系统

计量是计费前提。AI基础设施的计量对象包括 input token、output token、请求数、GPU数量、GPU时长、实例规格、存储容量、快照、公网流量、带宽峰值、专线、工单服务。

计量要准确、可追溯、可导出，否则客户无法信任账单。

3. 计费模型

常见计费方式有按量计费、套餐计费、包月包年、阶梯价、预付费、后付费、项目制、年运维费。

API适合按token或套餐；GPU适合按小时/天/月；私有化适合项目费+年服务费；专属推理实例适合实例费+并发或token费用。

4. 成本与毛利

API 成本包括上游模型、输入输出 token、工具调用、检索、重试、日志、审核、客服和运维。GPU 成本包括服务器折旧、GPU折旧、电费、机柜费、带宽费、制冷、运维、备件和空置率。

经营上要看客户收入减去真实成本后的毛利，而不是只看销售收入。

5. 账户权限和预算

企业客户需要企业空间、项目空间、部门、成员角色、管理员、开发者、财务、审计、API Key 权限、项目预算、单Key额度、每日消费上限、余额预警、欠费停用。

企业客户买的不只是模型能力，还买可管理性。

6. 交付形态

交付可以是公有API、企业API网关、GPU云平台、专属推理服务、私有化部署、一体机或机柜级方案。

交付越重，合同、部署、运维、培训、SLA、合规和售后要求越高。

7. SLA 和工单

SLA 是服务等级协议，常涉及可用性、响应时间、恢复时间、故障通知、服务补偿。工单等级通常分 P0/P1/P2/P3，分别对应大面积不可用、核心业务中断、部分功能异常、普通咨询。

SLA 不是一句“我们很稳定”，而是可量化承诺和故障处理机制。

8. 合规和风控

合规涉及生成式AI服务、AIGC标识、数据安全、个人信息保护、IDC/云资源业务边界、等保、安全审计、日志留存。风控涉及API Key泄露、盗刷、异常调用、内容安全、GPU滥用、租户隔离。

合规不是最后补丁，而是企业客户采购前就会问的基础问题。

计费对象

Token输入输出和上下文
请求数API调用次数
GPU时长GPU数量乘以运行时间
存储模型、数据、快照、日志
带宽公网流量和专线
服务SLA、运维、私有化支持

交付形态

公有API轻量调用
企业网关多模型统一管理
GPU云算力租赁
专属推理稳定API服务
私有化本地部署
一体机硬件软件整体交付

9. 基础补充：为什么运营层很重要

没有运营层，客户用了多少说不清，账单算不清，故障责任说不清，成本毛利看不清，合规风险控不住。运营层不是辅助功能，而是 AI 基础设施能不能商业化的核心。

10. 计费系统要能解释账单

客户看到一笔费用后，应该能追溯到具体项目、API Key、模型、时间、输入输出 token、GPU实例、存储、带宽。不能解释的账单会降低客户信任。

11. SLA 不能随便承诺

承诺 99.9% 或 99.99% 之前，要确认底层架构是否支持。单机部署、无备用模型、无故障切换、无监控告警，就不应该承诺很高可用性。SLA 是工程能力和合同责任的结合。

12. 合规要前置设计

如果后期才补合规，可能会影响产品结构。比如日志留存、数据隔离、AIGC标识、模型备案展示、内容审核、权限审计，这些都应该在平台设计阶段预留。

13. 商业模式要看真实成本

API业务要看上游模型成本、失败重试、内容审核和客服成本；GPU业务要看折旧、电费、机柜、带宽和空置率；私有化要看交付人力、现场实施、售后和版本维护。

客户最常问的运营问题

账单能不能按部门、项目、成员、API Key 分摊费用？
权限能不能限制员工调用模型和消费额度？
安全数据会不会被训练？日志能不能脱敏？
SLA服务挂了多久响应？能不能赔付或补偿？
合规有没有备案、等保、审计、AIGC标识能力？
Module 07

模块七：综合架构与学习路线总复盘

把所有模块合成一张完整 AI 基础设施认知图。

这一模块不是新增一个孤立知识点，而是把前面所有模块串成一条完整链路：客户需求、模型能力、API调用、推理部署、GPU算力、集群调度、IDC机房、计费运营、安全合规。
本模块目录
  1. 1. 从整体看 AI 基础设施
  2. 2. 六层架构
  3. 3. 五条核心流
  4. 4. 客户需求判断
  5. 5. 售前诊断问题
  6. 6. 你需要形成的能力
  7. 7. 建议沉淀的学习笔记
  8. 8. 最终记忆

1. 从整体看 AI 基础设施

AI基础设施不是模型、GPU、服务器或机房中的某一个点，而是一条完整服务链。客户看到的是API、控制台、账单和SLA；技术团队看到的是模型、推理框架、GPU集群和监控；机房团队看到的是电力、制冷、网络和运维；经营团队看到的是成本、毛利、利用率和续费。

2. 六层架构

第一层：客户与应用层。第二层：API / Token / 聚合平台。第三层：模型与推理部署。第四层：GPU服务器与集群调度。第五层：IDC机房与物理资源。第六层：计费运营、SLA、安全合规。

这六层不是孤立，而是从客户需求一路穿透到底层资源。

3. 五条核心流

能力流：机房 → GPU → 模型 → API → 客户。
调用流：请求 → 鉴权 → 限流 → 计量 → 路由 → 返回 → 扣费。
部署流：模型 → 框架 → GPU → 镜像 → 服务 → 网关。
成本流：电力、机柜、带宽、GPU折旧、API成本、运维 → 客户价格。
故障流：任何一层故障都会向客户体验传导。

4. 客户需求判断

客户说“我要AI”并不代表同一个需求。可能是买API、做知识库、部署模型、租GPU、做私有化、做企业AI网关。

你要先判断：数据是否敏感？是否需要本地部署？并发多少？预算多少？需要训练还是推理？是否要国产模型？是否要SLA？

5. 售前诊断问题

场景：做问答、客服、知识库、Agent、训练还是推理？
模型：需要闭源、国产、开源还是自有模型？
部署：公有API还是私有化？
数据：是否敏感，是否能出网？
并发：日调用量、峰值QPS、响应要求？
成本：按token、按GPU、按项目还是套餐？
合规：是否涉及备案、AIGC标识、等保、审计？

6. 你需要形成的能力

不是一开始就成为算法工程师、运维工程师或机房专家，而是要具备跨层理解能力：能听懂技术团队说什么，能问对客户需求，能画出整体架构，能看懂成本和毛利，能知道交付风险在哪里。

7. 建议沉淀的学习笔记

建议最终沉淀：AI基础设施总架构图、API调用链路图、Token计费表、模型部署链路图、GPU规格对比表、云平台资源生命周期图、IDC系统组成图、计费模型表、SLA指标表、合规能力清单、客户需求诊断表。

8. 最终记忆

客户要使用AI → 需要模型能力 → 模型通过API提供服务 → API通过token和资源用量计费 → 聚合平台统一管理模型和客户 → 自有模型需要推理部署 → 推理需要GPU服务器 → 多台服务器需要集群调度 → 服务器需要IDC机房承载 → 所有资源都要进入运营、SLA和合规系统。

客户需求到方案

只要调用模型API / Token套餐
多个业务统一用模型聚合平台 / 企业网关
要部署开源模型推理服务 + GPU
要训练或微调GPU云 / 裸金属
数据不能出内网私有化部署
要长期稳定高并发专属推理实例 + SLA

9. 基础补充：怎么判断自己真的懂了

如果你能把客户一句“我们想做AI”拆成：调用模型还是部署模型？公有云还是私有化？买token还是买GPU？训练还是推理？是否要SLA和合规？那说明你的整体框架已经建立。

10. 以后看项目方案的顺序

先看客户场景，再看数据和合规要求，再看模型能力，再看部署方式，再看算力资源，再看计费方式，最后看运维和SLA。不要一上来就讨论买什么GPU或用什么模型。

11. 你可以形成的核心表达

“AI基础设施是把模型能力、算力资源、机房资源、平台软件和运营合规连接起来的服务体系。客户买到的不只是模型，而是可调用、可管理、可计费、可保障、可合规的AI能力。”

12. 三类典型方案

轻量方案：聚合API + Token套餐。中型方案：企业AI网关 + 权限账单 + 多模型路由。重型方案：私有化部署 + GPU集群 + IDC资源 + 运维SLA + 合规交付。

13. 最重要的学习方法

不要孤立背术语。每学一个词，都问三件事：它解决什么问题？它在链路中连接谁和谁？如果它出问题，客户会感受到什么？这样你会更快建立基础设施思维。
